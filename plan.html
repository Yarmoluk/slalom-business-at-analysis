Week-by-Week Execution Plan

Week 1: Foundation — Smart Router + PII Scanner

Days 1-7 | Deploy two middleware services between UI and model APIs

Day 1-2: PII Scanner Deployment

Deploy the PII detection middleware as the first layer every prompt passes through. Scans for SSNs, credit cards, API keys, medical records, emails, and phone numbers. Blocks or warns before any text reaches a model. Enforces data classification tiers: public, internal, confidential, restricted. Blocks DeepSeek and open-source models from receiving anything above "public" tier.

CODE: code/pii_scanner.py | 12 PII patterns | 4 data tiers | Model allowlists per tier

Day 3-4: Smart Router Deployment

Replace the static Model Advisor with an intelligent routing engine. Auto-selects the optimal model based on: task type (reasoning, code, creative, classification, translation), input language, file size, cost tier preference, and tool calling requirements. Scores all 30+ models and returns the best match + 3 alternatives. Employees no longer need to manually choose from 30+ models.

CODE: code/smart_router.py | 15+ model registry | Scoring algorithm | Language & file size filtering

Day 5-7: Usage Analytics + Integration

Every model interaction logged: user, model, task type, tokens, cost, team, data tier. Dashboard endpoints for usage by model, team, provider, and cost. Wire the router and scanner into the existing business@ API layer (v1.0.455). This is the visibility layer Slalom currently lacks entirely.

CODE: smart_router.py /log-usage + /analytics/summary endpoints

Week 2: Knowledge — RAG Layer + Document Indexing

Days 8-14 | Make every model Slalom-aware

Day 8-10: RAG Service Deployment

Deploy the retrieval-augmented generation service with permission-trimmed search. Index Summit Methodology docs, proposal templates, and engagement patterns. Every prompt can now be augmented with relevant Slalom institutional knowledge before hitting any model. Role-based access control: consultants see "all" docs, partners see everything, restricted docs require approval.

CODE: code/rag_layer.py | Permission-trimmed search | Auto-augmented prompts | Role hierarchy

Day 10-12: Document Ingestion Pipeline

Build the indexing pipeline to pull from existing Slalom knowledge stores: SharePoint (methodology docs), Salesforce (engagement summaries), Confluence (delivery templates). Each document tagged with source type, permissions, and metadata. Batch indexing endpoint handles bulk import.

CODE: rag_layer.py /index + /index/batch endpoints | Source: summit, proposal, sow, template, engagement

Day 13-14: Seed Knowledge Base + Validate

Pre-load foundational documents: Summit Methodology overview, AI Transformation Framework (with Slalom's own 2026 research stats), proposal templates, standard SOW sections. Validate retrieval quality with test queries. Tune relevance scoring.

CODE: rag_layer.py SUMMIT_SEED_DOCS | 3 seed documents included | Extensible to hundreds

Week 3: Agents — Orchestrator + Pre-Built Pipelines

Days 15-21 | From single-turn chat to multi-agent workflows

Day 15-17: Agent Orchestrator Core

Deploy the multi-agent pipeline engine. Chains multiple agents in sequence, each with its own role (Analyzer, Researcher, Writer, Reviewer, Synthesizer), model selection, RAG context injection, and PII scanning at every boundary. The orchestrator calls the router, scanner, and RAG layer automatically at each step. No manual model selection needed.

CODE: code/agent_orchestrator.py | 5 agent roles | Step dependency resolution | Full service mesh integration

Day 17-19: RFP Response Pipeline (5 agents)

First production pipeline: Analyze RFP → Search past proposals (RAG) → Draft response sections → Brand/legal review → Final synthesis. Five agents chained in sequence, each using the optimal model for its role. The RFP Analyzer extracts requirements as JSON. The Researcher searches past proposals (permission-trimmed). The Writer drafts using Claude Sonnet 4.5. The Reviewer checks brand compliance. The Synthesizer produces the final deliverable on Opus 4.5.

CODE: agent_orchestrator.py PIPELINES["rfp-response"] | 5 steps | Replaces existing GPT-4o Digital Expert

Day 19-21: Client Discovery + SOW Generator Pipelines

Client Discovery (3 agents): Brief → Market Research (RAG) → Discovery Brief. Produces a 2-3 page partner-ready brief from a single paragraph of client context.

SOW Generator (3 agents): Scope Intake → SOW Writer (RAG + Sonnet 4.5) → Legal Review. Generates a complete Statement of Work with standard Slalom sections, leaving [PLACEHOLDER] for rates/dates.

CODE: agent_orchestrator.py PIPELINES["client-discovery"] + PIPELINES["sow-generator"]

Week 4: Polish — Dashboard, Deprecation, Launch

Days 22-30 | Production hardening and rollout

Day 22-24: Analytics Dashboard UI

Build a visual dashboard on top of the /analytics/summary endpoint. Show: model usage distribution (pie chart), cost by team (bar chart), adoption over time (line chart), top users, PII block rate, and data tier compliance. Deploy as a new tab in the business@ UI alongside Chat, Prompt Library, and Digital Experts.

DATA SOURCE: smart_router.py /analytics/summary | Build with Chart.js or Recharts

Day 24-26: Model Deprecation + Prompt Library Opening

Deprecate: Claude 3.7 Sonnet (superseded by 4/4.5), GPT-4o-mini (superseded by 5-mini), Llama 3 8B (250KB file limit, no tools), o1 (superseded by o3/o4-mini). Cut catalog from 30+ to ~20. Show deprecation warnings with migration paths.

Open Prompt Library: Enable employee submissions with a review queue. Tag by vertical, task type, and model. This unlocks 11,000 consultants' institutional knowledge.

Day 26-28: Upgrade Digital Experts to Frontier Models

Migrate all 6 existing Digital Experts from GPT-4o/4o-mini to current models:
- RFP Analyzer → Claude Sonnet 4.5 (better at structured extraction)
- Customer Story Generator → GPT-5.1 (better at creative narrative)
- Summit Advisor → Claude Opus 4.5 + RAG layer (methodology retrieval)
- Brand Reviewer → GPT-5-mini (fast, accurate, cost-effective)
- Digital Expert Architect → Claude Sonnet 4.5 (better system prompt generation)
- Connexus Advisor → Smart Router integration (auto-routes instead of manual)

Day 29-30: Production Launch + Documentation

Deploy all 4 services via Docker Compose. Run load testing. Write internal announcement and training guide. Launch to pilot group (50 consultants in Minneapolis market), measure adoption for 2 weeks, then expand.

CODE: code/docker-compose.yml + code/Dockerfile | Single `docker compose up` deployment

Investment & ROI

Engineering Effort

  • 2 engineers, 30 days (320 person-hours)
  • All 4 services: ~1,000 lines of Python total
  • FastAPI + Docker — standard Slalom tech stack
  • No model training or fine-tuning required
  • Integrates with existing business@ API (v1.0.455)

Expected Impact

  • RFP response time: days → hours (5-agent pipeline)
  • Model selection time: minutes → zero (auto-routing)
  • PII exposure risk: unknown → measured and blocked
  • Cost visibility: blind → per-model, per-team tracking
  • Knowledge reuse: 0% → every prompt RAG-augmented

Risk Mitigation

  • All services are middleware — no changes to existing models or APIs
  • Docker Compose = one command deploy, one command rollback
  • Pilot with 50 Minneapolis consultants before org-wide
  • Each week is independently deployable — value from Day 3
  • If any service fails, business@ continues to work as-is
